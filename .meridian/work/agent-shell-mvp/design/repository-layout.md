# Repository Layout

> Resolves Q3 from `requirements.md`. Defines where the new shell code lives in
> meridian-channel, how launching the shell relates to the existing CLI, what
> `uv sync` installs, what `mars sync` ships, and how Dad actually starts the
> thing. Read the design [overview](./overview.md) first; this doc is the
> "where on disk" companion to it.

Two things to notice. **One**, there is already a `src/meridian/server/` —
it hosts the small MCP server. The new agent-shell backend is conceptually a
sibling, not a replacement. **Two**, every Python concern lives under one
hatchling-built package; there is no `backend/` sibling and no second
distribution. We will preserve that property.

## 2. Decision: backend lives inside the meridian package

**Chosen:** new code lives under `src/meridian/shell/`. Not a sibling
`backend/` directory, not a second distribution.

**Options considered**

| Option | Where | Verdict |
|---|---|---|
| **A. Sibling `backend/`** | `backend/` next to `src/` | Rejected. Two Python projects, two `pyproject.toml`s, two editable installs, two import roots. Means the CLI cannot trivially `from meridian.shell import ...` and the shell cannot trivially `from meridian.lib.catalog import ...`. The seam adds friction with no payoff. |
| **B. Sibling `meridian-shell/` source submodule** | parallel to `meridian-base/` | Rejected. Source submodules are for `.agents/` content, not Python source. Wrong tool. |
| **C. Inside the meridian package, `src/meridian/shell/`** | new package | **Chosen.** One `uv sync`, one editable install, one import namespace. CLI subcommands can import shell internals trivially. The harness abstraction (§3 of overview) sits next to the existing single-shot adapters in `lib/harness/`, which is exactly where shared mechanism belongs. |

**Rationale.** The shell *is* meridian now (Decision 1 in requirements). The
package gets bigger. That is fine: the cost of a fatter package is one bounded
subtree; the cost of two packages is permanent integration friction. SOLID
applies inside the package — `shell/` is one bounded subtree, the existing
`cli/`, `lib/state/`, `lib/catalog/`, `lib/launch/` stay unchanged. Importing
across the seam (`shell` → `lib.catalog`) is allowed and expected; importing
back (`lib.catalog` → `shell`) is forbidden by convention and enforced by
review.

## 3. Decision: frontend lives at top-level `frontend/`

**Chosen:** copy frontend-v2 to `frontend/` at the repo root, sibling to
`src/`, `tests/`, `scripts/`. Keep its own `package.json`, `pnpm-lock.yaml`,
`vite.config.ts`, `tsconfig.json`, Storybook config.

**Options considered**

| Option | Where | Verdict |
|---|---|---|
| **A. `src/meridian/shell/frontend/`** | inside Python package | Rejected. Python wheels should not ship a 200 MB Node project. Hatchling would need explicit excludes. Toolchains do not mix. |
| **B. `frontend/` at repo root** | sibling to `src/` | **Chosen.** Clear toolchain boundary. Vite/pnpm own their tree. Hatchling never sees it. CI splits cleanly into a Python job and a Node job. |
| **C. Standalone repo** | external | Rejected. Decision 1 says copy *into* this repo, not symlink, not cross-repo. Cross-repo coupling is the exact thing we are escaping. |

**What gets copied.** A snapshot of `../meridian-flow/frontend-v2/` into
`frontend/`. From there it evolves independently — drift from the
meridian-flow original is expected and acceptable. The `frontend-integration`
doc enumerates what stays vs cuts vs new (writing-app legacy is mostly cut;
activity stream + threads + ws-client + ui atoms stay).

**Bundling.** `pnpm build` produces `frontend/dist/`. Releases ship that
directory prebuilt. The Python shell server serves it as static assets.
`pnpm` is a development requirement, not a runtime requirement for Dad's V0
validation path.

## 4. Decision: launching is `meridian shell` subcommand

**Chosen:** the shell is launched as `meridian shell start`, a new top-level
subcommand of the existing CLI. Not a separate binary.

**Options considered**

| Option | Verdict |
|---|---|
| **A. Separate binary `meridian-shell`** | Rejected. Two install targets, two entry points in `pyproject.toml`, two `--help` discoveries. Adds install complexity and a parallel config story. |
| **B. `meridian shell` subcommand** | **Chosen.** One entry point. Reuses existing cyclopts argparse plumbing. Inherits config precedence (CLI flag > env > YAML > project > user > harness default — see CLAUDE.md). Discoverable via `meridian --help`. Sits next to `meridian spawn`, `meridian work`, `meridian session`, `meridian mars`, all of which the shell needs to talk to anyway. |
| **C. `meridian server` subcommand** | Rejected. The MCP server already lives at `src/meridian/server/`. Naming collision and conceptual confusion. `shell` is the right verb because the artifact IS a shell around an agent harness. |

**Subcommand surface.**

```text
meridian shell start         # boot backend + serve frontend + open browser
meridian shell dev           # dev mode: vite dev server + backend with reload
meridian shell stop          # stop a running shell (looks up PID file)
meridian shell status        # report whether a shell is running, on what port
meridian shell doctor        # check venv, frontend bundle, kernel, port
```

`start` is the only one Dad ever runs. The rest are for development and
debugging.

## 6. mars sync surface for the shell

The shell ships **one new source submodule**: `meridian-shell-agents/`,
sibling to `meridian-base/` and `meridian-dev-workflow/`.

It contains:

- `agents/data-analyst.md` — the data-analyst agent profile (Decision 1 of
  validation, biomedical V0)
- `skills/biomedical-analyst/SKILL.md` — biomedical analysis methodology
- `skills/interactive-tool-authoring/SKILL.md` — how to write a new
  interactive tool that the shell can call (PyVista point picker, etc.)

It is registered in `mars.toml` exactly like the other two source submodules.
The canonical authoring workflow stays unchanged from CLAUDE.md:

1. Edit in the source submodule (e.g.
   `meridian-shell-agents/skills/biomedical-analyst/SKILL.md`)
2. Commit and push the source repo change
3. Update package refs if needed with `meridian mars add ...`
4. Run `meridian mars sync` to regenerate `.agents/`

The shell loads agents and skills out of `.agents/` via the existing
`lib/catalog/agent.py`, `lib/catalog/skill.py`, `lib/launch/resolve.py`,
`lib/launch/prompt.py` machinery. **No parallel registry, no shell-specific
agent format.** This is the seam where the biomedical persona attaches to a
generic shell — see [agent-loading.md](./agent-loading.md) for how the
resolved profile + skills feed into the long-lived Claude Code adapter via
`--append-system-prompt` (profile + skills merged) and `--mcp-config`
(shell-owned tools).

**Domain pivot story.** Pivoting away from biomedical = ship a new source
submodule (e.g. `meridian-legal-agents/`) with its own agents + skills +
interactive tools, register it in `mars.toml`, run `meridian mars sync`. The
shell does not change. This is the OCP commitment from Decision 6.

## 7. uv sync story

`pyproject.toml` grows in one place for shell-core dependencies. Heavy
biomedical packages do **not** live here.

**Base dependencies** (everyone who installs meridian-channel pays):

```toml
dependencies = [
  # existing
  "coolname>=4.0.0",
  "cyclopts>=3.0",
  "mars-agents==0.0.9",
  "mcp>=1.0",
  "pydantic>=2.12",
  "pydantic-settings>=2.13",
  "python-frontmatter>=1.1",
  "structlog>=25.0",
  # new for shell core
  "fastapi>=0.115",
  "uvicorn[standard]>=0.32",
  "websockets>=13",
  "jupyter_client>=8.6",
  "ipykernel>=6.29",
]
```

The shell core deps are small, pure-Python (or near enough), and useful even
to dev-orchestration users for testing. They go in base.

The biomedical runtime is provisioned separately from a manifest at
`src/meridian/shell/runtime/manifests/biomedical.toml` by
`meridian shell init biomedical`.

**Install paths.**

```bash
uv sync                        # CLI + shell core
uv sync --extra dev            # developer tooling
meridian shell init biomedical # provision ~/.meridian/venvs/biomedical/
```

**Why split.** Dev-orchestration users (the existing audience) should not be
forced to download SimpleITK and PyVista. Biomedical V0 users get a dedicated
domain runtime. Future domains add their own manifests (`legal`, `finance`,
etc.) without bloating the base project environment.

## 8. Frontend install + build

`frontend/` is a self-contained pnpm workspace. Development workflow:

```bash
cd frontend
pnpm install
pnpm build           # produces frontend/dist/
```

`meridian shell start` does **not** build the frontend in V0. Releases and
developer-mediated local installs ship `frontend/dist/` prebuilt. If `dist/`
is missing, start fails loudly and points the developer at `pnpm build`.

**Dev mode.** `meridian shell dev` does *not* serve `frontend/dist/`. Instead
it spawns `pnpm dev` (vite dev server) on a separate port and configures the
backend's static route to proxy unmatched paths to vite. Hot reload works for
both Python (uvicorn `--reload`) and React (vite HMR).

**Pnpm vs npm.** frontend-v2 already uses pnpm (per `pnpm-lock.yaml` in the
grounding report). We keep pnpm. Adding npm support is not on the V0 roadmap.

## 9. `meridian shell start` semantics

What `meridian shell start` does, in order:

1. **Resolve config.** Read CLI flags, env, profile YAML, project config
   (CLAUDE.md precedence). Determine: port, work item, agent profile, browser
   open behavior.
2. **Verify runtime venv.** Check the biomedical runtime exists at
   `~/.meridian/venvs/biomedical/` when the selected profile needs it. If
   missing, fail loudly with `meridian shell init biomedical`.
3. **Verify frontend bundle.** If `frontend/dist/` is missing, fail loudly and
   point the developer at `pnpm build`. V0 runtime does not depend on `pnpm`.
4. **Verify mars sync.** If `.agents/` is older than `mars.lock`, warn and
   suggest `meridian mars sync`. Do not auto-run — mars is the user's source
   manifest, not ours to mutate.
5. **Acquire port.** Default 8765. If in use, fail with `--port` suggestion
   unless `--auto-port` was passed.
6. **Write PID file.** `.meridian/shell.pid` (atomic tmp+rename, per
   files-as-authority discipline). Includes port, started-at, work item.
7. **Boot uvicorn.** Start FastAPI app on the port. The app brings up
   `SessionManager`, `EventRouter`, `FrontendTranslator`. Harness adapters
   are lazy — the Claude Code subprocess only spawns when the first browser
   client subscribes.
8. **Open browser.** Unless `--no-browser`. Default to system browser at
   `http://localhost:<port>/`.
9. **Wait.** Run until SIGINT/SIGTERM. On shutdown, gracefully stop the
   harness subprocess, close the kernel, remove the PID file.

**CLI ergonomics.**

```text
meridian shell start [OPTIONS]

Options:
  --port INTEGER             Bind port (default: 8765)
  --auto-port                If port in use, pick next free
  --no-browser               Do not open the browser
  --dev                      Dev mode: pnpm dev + uvicorn --reload
  --profile TEXT             Agent profile name (default: from project config)
  --work-item TEXT           Work item id (default: most recent)
  --harness TEXT             Harness adapter (default: claude-code; V0 only)
  --no-frontend-build        Skip the frontend bundle staleness check
  --log-level TEXT           uvicorn log level (default: info)
```

`--dev` is sugar for `meridian shell dev`. Both forms exist; `meridian shell
dev` is the documented form.

## 10. CI impact

V0 trusts manual verification (smoke tests under `tests/smoke/`). V1 adds:

- **Python job.** Existing `uv run ruff check .` + `uv run pyright` + `uv
  run pytest-llm`. Now also covers `src/meridian/shell/`. Pyright must stay
  at zero errors.
- **Frontend job (new).** `cd frontend && pnpm install --frozen-lockfile &&
  pnpm typecheck && pnpm lint && pnpm build`. The built `dist/` artifact is
  archived for release packaging. Cached on `pnpm-lock.yaml`
  hash. Runs in parallel with the Python job.
- **Shell smoke job (V1).** Boots `meridian shell start --no-browser` against
  a fixture work item, hits the WebSocket, asserts a `RUN_FINISHED` event
  comes back. Headless, scripted, no real harness — uses a `FakeHarnessAdapter`
  to keep the test harness-independent.

V0 ships without the shell smoke job. The OCP commitment to the harness
abstraction is preserved by a single unit test that asserts a fake adapter
satisfies the protocol — see `harness-abstraction.md`.

## 11. Coexistence with existing meridian CLI

The shell is **purely additive**. Every existing subcommand keeps working
exactly as it does today:

```text
meridian spawn ...        # dev orchestration — unchanged
meridian work ...         # work item lifecycle — unchanged
meridian session ...      # session transcript reads — unchanged
meridian mars ...         # package management — unchanged
meridian config ...       # config inspection — unchanged
meridian doctor ...       # health check — unchanged (now also scans shell.pid)
meridian models ...       # model catalog — unchanged
meridian report ...       # report mgmt — unchanged
meridian shell ...        # NEW
```

Dev-orchestration workflows are unaffected. Agents that spawn other agents
via `meridian spawn` continue to ignore the shell entirely. The shell is one
subcommand sitting next to the others, not a takeover.

**Shared state.** Both surfaces read and write `.meridian/`. The shell's
session-lived harness subprocess writes its own session log under
`.meridian/sessions/`, exactly like a spawn would. `meridian session log`
can read shell sessions transparently — they are not a second class. The
shell's work item directory is a normal `.meridian/work/<id>/`.

**No new state root.** Resist the temptation to give the shell its own
`.meridian/shell/` subdirectory. Files-as-authority means one source of
truth, not one per surface.

## 12. Edge cases

These are the failure modes the start path must handle deliberately, not
discover at 2 AM in a Yao Lab basement.

| Edge case | Handling |
|---|---|
| **User runs `meridian shell start` without building frontend** | Fail fast. V0 runtime expects shipped `frontend/dist/`. Developer runs `pnpm build` or uses `meridian shell dev`. |
| **`pnpm` not on `$PATH`** | Only matters in development. `meridian shell dev` fails with install instructions; production start still works if `frontend/dist/` exists. |
| **Conflicting Python dep versions** (e.g. user has scipy 1.10 globally) | uv venv isolates this. The shell only ever runs against the project venv. If the user has bypassed uv, doctor reports it. |
| **Biomedical runtime missing for `data-analyst` profile** | Step 2 fails loudly with exact `meridian shell init biomedical` command. Do not auto-install. |
| **Port in use** | Fail with `--port N` suggestion. With `--auto-port`, scan upward from default until free. PID file records the actual port. |
| **Stale PID file from prior crash** | On start, read PID file, check if PID is alive. If not, log a warning and overwrite. Crash-only design: recovery IS startup. |
| **Two `meridian shell start` invocations race** | `fcntl.flock` on the PID file. Second invocation fails fast with the running port. |
| **Shell crashes leaving orphaned kernel** | `meridian doctor` scans for orphan jupyter kernels associated with stale PID files and offers to reap them. Same pattern as the existing spawn orphan reconciler. |
| **Shell crashes leaving orphaned harness subprocess** | Same — doctor scans, offers to reap. The session log already records the subprocess PID. |
| **`mars sync` removes a profile the running shell depends on** | Shell is reading `.agents/` lazily. On the next session start it fails with "agent profile X no longer in catalog." Existing sessions keep running because the profile + skills were already injected at session start (Decision 6 about resilience). Warn the user; do not auto-restart. |
| **Frontend dist older than source after a `git pull`** | Start warns in developer environments; releases ship a fresh `dist/` bundle. `meridian shell dev` is the iteration path, not runtime auto-build. |
| **`.agents/` is newer than `mars.lock`** | Means user hand-edited `.agents/`. Per CLAUDE.md, this is forbidden. Doctor warns; shell start warns; neither blocks. |
| **Browser blocked from auto-open (headless host, SSH)** | `--no-browser` is the explicit form. Auto-open failures are non-fatal — log the URL and continue running. |
| **Dad closes the browser tab** | Backend keeps running for the 30-second reconnect window, then idles until explicitly stopped. He reopens the same localhost URL and gets back to the same work item. Closing the terminal that ran `meridian shell start` *does* stop the backend; V0 install + launch is developer-mediated, not launcher-driven. |
| **Long-running analysis blocks SIGTERM** | Shutdown sends SIGTERM to the harness subprocess, waits 5s, then SIGKILL. The persistent kernel gets a graceful shutdown via `jupyter_client`'s `shutdown_kernel()`. PID file is removed last. |
| **Permission denied on port < 1024** | Default port 8765 sidesteps this. If user passes `--port 80`, fail with a clear "use a port ≥ 1024 or run as root" message. |

## 13. What this doc commits to

- Shell backend lives at `src/meridian/shell/`, inside the meridian package.
- Frontend lives at `frontend/`, sibling to `src/`, with its own toolchain.
- Shell is launched via `meridian shell start`, a new CLI subcommand. No
  separate binary.
- Shell agents and skills ship as a new source submodule
  `meridian-shell-agents/`, regenerated into `.agents/` via `meridian mars sync`.
- Heavy biomedical deps live in a separate runtime manifest and venv. The base
  install stays small.
- The shell is purely additive to the existing CLI. No existing subcommand
  changes behavior.
- One PID file (`.meridian/shell.pid`), one port, one work item per running
  shell instance.
- Crash-only discipline: PID file is the truth; recovery is startup; doctor
  reaps orphans.
