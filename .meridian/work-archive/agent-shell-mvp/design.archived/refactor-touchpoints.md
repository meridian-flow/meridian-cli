# Refactor Touchpoints

**Source**: explorer pre-pass p1160 (2026-04-08)
**Scope**: files and modules that must change (or be newly introduced) to implement the D36/D37 refactor in meridian-channel.

## How To Read
1. Start with the harness table first; that is where the AG-UI event mapping and control protocol live.
2. Treat the status column as the implementation signal: `must change`, `may change`, or `unchanged`.
3. Read the `Current role` cell for what the file does today, not what it should become.
4. Read `Refactor impact` for the smallest likely change boundary, not a design proposal.
5. `Consumers` lists the main importers or call sites that will feel the ripple.
6. `Risk` calls out the most likely regression path: CLI surface, output shape, persistence, or dogfood workflow.
7. The "Adjacent load-bearing" section includes launch and ops files outside the requested directories because they are on the critical path.
8. The structural analysis answers the placement questions the architects need before they draw new APIs.
9. The tests and smoke sections are a regression map, not a test plan.
10. The unchanged list at the end is intentionally short: only files the explorer is confident the refactor should not touch.

## Structural Analysis
- **AG-UI event emitter** should live in a new harness sibling module, not in `transcript.py`. `transcript.py` is post-hoc text normalization; AG-UI is a wire-format concern. A new sibling such as `harness/ag_ui_events.py` or `harness/events.py` is the cleaner home.
- **Stdin control reader** should also live in `harness/`, either as a small shared `control_channel.py`/`stdin_control.py` or as thin adapter-local shims over a shared frame model. The adapters do not share the same runtime semantics, so do not bury this in `common.py`.
- **Streaming launch type** belongs in `launch_types.py` unless it becomes too large. It is launch-policy metadata, not a state-store concern and not a CLI concern.
- **`spawn inject <spawn_id>`** should resolve the target through the per-spawn log/state directory first, not through the artifact store. Today the only authoritative per-spawn anchor is `.meridian/spawns/<id>/` from `paths.py`; `spawn_store.py` currently has no live-control metadata to point at a stdin handle or FIFO.
- **Durable control path**: if the refactor introduces a FIFO/socket/file path for live control, that path needs to be stored in spawn state and discoverable by `spawn_store.py` and `paths.py`. Without that, inject cannot be replayed or inspected after a crash.
- **Dogfood workflow dependencies**: the current dogfood workflow still depends on `report.md`, `output.jsonl`, `stderr.log`, `prompt.md`, `params.json`, `heartbeat`, and `harness.pid`. `spawn show`, `spawn log`, `spawn wait`, `spawn stats`, `--from`, and `--fork` all assume those artifacts or their projections exist.
- **`reaper.py` sensitivity**: treats `report.md` as a durable-completion signal and also watches the existing stdout/stderr files for freshness. If streaming mode changes where completion appears, reaper policy must follow.
- **`launch/stream_capture.py` bridge**: already has an event-observer callback, which is useful, but it is still generic JSONL framing. That makes it a good bridge, not the place to design the AG-UI taxonomy itself.

## Summary
- **Total load-bearing files identified**: 37.
- **Top 3 highest-risk touchpoints**: `harness/claude.py`, `harness/codex.py`, `harness/opencode.py` — all three adapters must gain AG-UI emission and a control-channel surface while preserving every existing artifact/report contract the dogfood workflow depends on.
- **Surprises / structural concerns for the architects**:
  1. `state/spawn_store.py` has **no live-control metadata today**, so the control-handle path (`.meridian/spawns/<id>/control.sock` or similar) has to be designed into `paths.py` + `spawn_store.py` before `spawn inject` has anything to target.
  2. `state/reaper.py` **hardcodes `report.md`** as a durability signal and also treats `output.jsonl`/`stderr.log` freshness as liveness; the AG-UI stream must not silently invalidate either.
  3. `launch/stream_capture.py` already has a generic parsed-event observer callback — **a natural bridge, but not the home for AG-UI taxonomy**. Keep the taxonomy in the adapter layer, the plumbing in `stream_capture`.
  4. `ops/spawn/log.py` and `ops/session_log.py` both **parse harness-specific JSON shapes** for `spawn log` and `session log`. Transcript/compaction display is the main regression boundary where AG-UI translation could accidentally break dogfood.
  5. `launch/process.py` already **copies stdin straight into the child PTY** for primary launches — the streaming control channel must not collide with that path or it will break interactive `meridian` sessions.
