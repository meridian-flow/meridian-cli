# Harness Materialization

**Status:** approved

## Problem

When `meridian start` (or `meridian run spawn`) resolves an agent profile and its skills, those resources may live in locations the target harness can't read natively:

- Agent in `.agents/agents/primary.md` → Claude Code only reads `.claude/agents/`
- Skill in bundled package resources → no harness reads from there natively
- Skill in `.agents/skills/orchestrate/` → Claude Code doesn't read `.agents/`

Currently we work around this with ad-hoc JSON (`--agents '{...}'`), but that:
1. Doesn't let the harness natively resolve skill references inside the agent
2. Produces ugly commands in `--dry-run`
3. Doesn't survive harness-level agent resolution (skills won't load)

## Solution: Materialize into Harness-Native Dirs

Before launching, copy any agent/skill the harness can't find natively into the harness's own directory, scoped by chat ID to avoid collisions between concurrent sessions.

### Naming Convention

```
_meridian-<chat-id>-<original-name>
```

- **Primary launch**: `_meridian-c1-primary.md`, `_meridian-c1-orchestrate/` (full directory)
- **Subagent run**: `_meridian-c3-coder.md`, `_meridian-c3-researching/` (full directory)

Chat IDs (`c1`, `c2`, ...):
- Generated by `start_session()` before command building in all paths (primary, blocking run, background run)
- Unique per session within a space
- Short and filesystem-safe (alphanumeric)
- Already used for session locking — cleanup aligns with session lifecycle

## Harness Native Directory Map

Hardcoded per harness — these are fixed by each harness's spec:

```python
@dataclass(frozen=True, slots=True)
class HarnessLayout:
    """Directories a harness reads agents/skills from natively."""
    agents: tuple[str, ...]         # project-local agent dirs
    skills: tuple[str, ...]         # project-local skill dirs
    global_agents: tuple[str, ...]  # home-dir agent dirs
    global_skills: tuple[str, ...]  # home-dir skill dirs
    # First entry in agents/skills is the materialization target.

HARNESS_NATIVE_DIRS: dict[str, HarnessLayout] = {
    "claude": HarnessLayout(
        agents=(".claude/agents",),
        skills=(".claude/skills",),
        global_agents=("~/.claude/agents",),
        global_skills=("~/.claude/skills",),
    ),
    "codex": HarnessLayout(
        agents=(".agents/agents", ".codex/agents"),
        skills=(".agents/skills", ".codex/skills"),
        global_agents=(),
        global_skills=(),
    ),
    "opencode": HarnessLayout(
        agents=(".agents/agents", ".opencode/agents"),
        skills=(".agents/skills", ".opencode/skills"),
        global_agents=("~/.opencode/agents",),
        global_skills=("~/.opencode/skills",),
    ),
}
```

> `.agents/` is a cross-harness standard read natively by codex and opencode, but NOT by Claude Code.

The **first entry** in `agents`/`skills` is the materialization target directory.

## Prerequisite Refactor: AgentProfile raw_content

Current `parse_agent_profile` extracts known frontmatter fields into `AgentProfile` but **discards the raw file text**. Unknown frontmatter keys are lost, making faithful reconstruction impossible.

### Change to `AgentProfile`

```python
@dataclass(frozen=True, slots=True)
class AgentProfile:
    # ... existing fields unchanged ...
    raw_content: str   # full original file text (frontmatter + body)
```

### Change to `parse_agent_profile`

```python
def parse_agent_profile(path: Path) -> AgentProfile:
    markdown = path.read_text(encoding="utf-8")
    frontmatter, body = split_markdown_frontmatter(markdown)
    # ... existing field extraction unchanged ...
    return AgentProfile(
        # ... existing fields ...
        raw_content=markdown,   # NEW: store raw text
    )
```

Builtin profiles (hardcoded, `path=<builtin>`) set `raw_content=""`. The materialization module handles this case by reconstructing from known fields — acceptable since builtins are fully controlled by us.

## Materialization Flow

```
meridian start (space, chat_id=c1) / meridian run spawn (run, chat_id=c3)
│
├── 1. Resolve agent profile (from SearchPathConfig, bundled fallback)
│     → e.g. found in `.agents/agents/primary.md`
│     → agent has skills: [orchestrate, meridian-run]
│
├── 2. Resolve each skill (from SkillRegistry)
│     → e.g. orchestrate found in `.agents/skills/orchestrate/SKILL.md`
│     → e.g. meridian-run found in bundled package resources
│
├── 3. Determine target harness (from model routing)
│     → e.g. claude
│
├── 4. Check what's already native for this harness
│     For each skill, check ALL dirs in harness layout (local + global):
│       - `.claude/skills/orchestrate/SKILL.md` exists? → native, skip
│       - `~/.claude/skills/orchestrate/SKILL.md` exists? → native, skip
│       - neither → must materialize
│     Same for the agent profile.
│
├── 5a. ALL native (agent + all skills) → launch directly
│     claude --agent primary --append-system-prompt <context>
│
├── 5b. ANY skill or agent needs materializing →
│     ├── Cleanup old _meridian-c1-* from this chat's previous attempt (if any)
│     │
│     ├── Copy missing skill DIRECTORIES to harness native dir:
│     │   .claude/skills/_meridian-c1-orchestrate/   (full tree: SKILL.md, scripts/, references/)
│     │   .claude/skills/_meridian-c1-meridian-run/   (full tree)
│     │
│     ├── Create materialized agent file:
│     │   .claude/agents/_meridian-c1-primary.md
│     │   - Take raw_content from AgentProfile
│     │   - Rewrite skills: line in frontmatter via _rewrite_agent_skills():
│     │     native skills keep original name,
│     │     materialized skills use _meridian-c1-<name>
│     │   - If raw_content is empty (builtin), reconstruct from known fields
│     │
│     └── Launch: claude --agent _meridian-c1-primary --append-system-prompt <context>
│
└── 6. Cleanup: remove _meridian-<chat-id>-* files
      - On session stop (stop_session already called on all paths)
      - On cleanup_orphaned_locks / stale session cleanup
```

## Copy Semantics

### Skills: Full Directory Tree
Skills are directory-level resources. `orchestrate/` may contain:
```
orchestrate/
  SKILL.md
  scripts/
  references/
```
Materialization copies the **entire directory tree** via `shutil.copytree`, not just SKILL.md.

### Agents: Raw Content with Skills Rewrite
Agent materialization takes `AgentProfile.raw_content` and rewrites only the `skills:` line in frontmatter. Everything else (unknown keys, comments, body) is preserved exactly.

For builtin profiles (`raw_content=""`), reconstruct a minimal markdown file from known fields:
```markdown
---
name: <name>
model: <model>
skills: [<rewritten skills>]
sandbox: <sandbox>
---
<body>
```

## Dry-Run Behavior

Materialization is **skipped** for `--dry-run`. Instead:
- Compute what the agent name WOULD be (`_meridian-c1-primary` or `primary`)
- Show it in the dry-run command output
- Don't write any files to disk

## Safety Protocol

1. **Prefix `_meridian-<chat-id>-`**: All materialized files include the chat ID. Never touch files without the prefix.
2. **Scoped cleanup**: Only delete `_meridian-<my-chat-id>-*`, never other scopes.
3. **Re-materialize on every launch**: Delete old for this scope, write fresh. They're ephemeral.
4. **Never overwrite user files**: If `.claude/skills/orchestrate/` exists (user-created), it's native — don't touch it.
5. **Check both project AND home dirs**: A skill at `~/.claude/skills/foo/` counts as native for Claude.
6. **Gitignore**: Add `_meridian-*` patterns to harness dirs on first materialization.

## Implementation Steps

### Step 0: AgentProfile refactor (prerequisite)

File: `src/meridian/lib/config/agent.py`

1. Add `raw_content: str` field to `AgentProfile` dataclass
2. Store `markdown` in `parse_agent_profile` as `raw_content=markdown`
3. Set `raw_content=""` in `_builtin_profiles()`
4. Update all test files that construct `AgentProfile` directly — add `raw_content=""`
5. Run tests, verify no regressions

### Step 1: HarnessLayout dataclass + registry

New file: `src/meridian/lib/harness/layout.py`

- `HarnessLayout` dataclass
- `HARNESS_NATIVE_DIRS` registry
- `harness_layout(harness_id: str) -> HarnessLayout`
- `materialization_target_agents(layout, repo_root) -> Path`
- `materialization_target_skills(layout, repo_root) -> Path`

### Step 2: Materialization module

New file: `src/meridian/lib/harness/materialize.py`

```python
def is_skill_native(skill_name: str, layout: HarnessLayout, repo_root: Path) -> bool
def is_agent_native(agent_name: str, layout: HarnessLayout, repo_root: Path) -> bool

def _rewrite_agent_skills(raw_content: str, skill_mapping: dict[str, str]) -> str
def _reconstruct_builtin_agent(profile: AgentProfile, skill_mapping: dict[str, str]) -> str

@dataclass(frozen=True, slots=True)
class MaterializeResult:
    agent_name: str                       # name to pass to --agent
    materialized_agent: bool
    materialized_skills: tuple[str, ...]
    native: bool                          # everything already native?

def materialize_for_harness(
    agent_profile: AgentProfile | None,
    skills: tuple[SkillContent, ...],
    harness_id: str,
    repo_root: Path,
    chat_id: str,
    dry_run: bool = False,
) -> MaterializeResult

def cleanup_materialized(harness_id: str, repo_root: Path, chat_id: str) -> int
def cleanup_all_materialized(harness_id: str, repo_root: Path) -> int
```

Key implementation details:
- `materialize_for_harness` with `dry_run=True` computes result without writing
- Skill materialization uses `shutil.copytree` for full directory copy
- Agent materialization: if `raw_content` → `_rewrite_agent_skills`; if empty → `_reconstruct_builtin_agent`
- Creates target directories (`mkdir -p`) if they don't exist

### Step 3: Wire into launch.py (primary agent)

Replace `profile_is_native` / ad-hoc JSON / three-way branching with:
1. `chat_id` already generated via `start_session()` before command building
2. `result = materialize_for_harness(profile, skills, harness_id, repo_root, chat_id, dry_run=request.dry_run)`
3. `command = ["claude", "--agent", result.agent_name, "--append-system-prompt", prompt, ...]`
4. Remove: ad-hoc JSON path, `profile_is_native` logic, `import json` (if no longer needed)

### Step 4: Wire into run spawn (subagent)

In `_execute_run_blocking` and `_execute_background_run`:
1. `chat_id` is already generated via `start_session()`
2. Call `materialize_for_harness(...)` before spawn
3. Pass materialized agent name to harness command builder

### Step 5: Cleanup integration

- `stop_session()` call sites: add `cleanup_materialized(harness_id, repo_root, chat_id)` after stopping
- `cleanup_stale_sessions()`: already cleans stale session locks; add materialized file cleanup
- `cleanup_orphaned_locks()`: optionally sweep `_meridian-*` for sessions that no longer exist

### Step 6: Tests

- `_rewrite_agent_skills`: preserves unknown frontmatter, comments, body; rewrites only skill names
- `_reconstruct_builtin_agent`: generates valid markdown for builtins
- `is_skill_native` / `is_agent_native`: project dir, home dir, missing
- `materialize_for_harness`:
  - All native → no writes, original agent name
  - Mixed → only missing ones materialized, agent rewritten
  - All missing → everything materialized
  - Full skill directory tree copied (not just SKILL.md)
  - Agent raw_content preserved, only skills: rewritten
  - Builtin agent (raw_content="") reconstructed correctly
  - dry_run=True → no filesystem writes
  - Chat ID scoping: `_meridian-c1-*` names
- Cleanup:
  - Scoped cleanup removes only matching chat ID
  - Full cleanup removes all `_meridian-*`
  - Never removes non-prefixed files
- Integration:
  - `_build_interactive_command` uses materialized agent
  - Dry-run doesn't write files

## What This Replaces

- The ad-hoc JSON (`--agents '{...}'`) path in `_build_interactive_command`
- The `profile_is_native` boolean logic
- The three-way branching (native / ad-hoc / no-profile)

All collapse into: resolve → materialize if needed → `--agent <name>`.
